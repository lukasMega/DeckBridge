0 · Topology

The bridge forms a two-hop path between a desktop running Elgato Stream Deck software and a physical USB Stream Deck device. On the left, the desktop connects over TCP to the bridge using the CORA network protocol (a framed binary protocol Elgato uses for its Network Dock hardware). The bridge acts as a software-emulated dock: it listens on port 5343 (primary dock channel) and a second child port (5344 in our implementation, 20002 on real hardware), advertises itself over mDNS as _elg._tcp, and speaks the full CORA handshake including capabilities and keepalives. On the right side of the bridge, a dedicated USB HID worker thread opens the physical Stream Deck device via libhidapi, polls for key-state input reports every 5 ms, and writes chunked JPEG/BMP image data via output reports. The bridge translates between the two worlds in real time: CORA image chunks become hid_write calls; HID key-state changes become CORA key-event frames.

1 · Elgato Stream Deck over USB (HID)

The Stream Deck appears to the operating system as a standard USB HID device. All communication goes through three HID report types:

  • Input reports — device-to-host; button state delivered by polling with hid_read_timeout(5 ms).
  • Output reports — host-to-device; image pixels sent in chunked 1024-byte packets via hid_write.
  • Feature reports — bidirectional control channel; used for brightness, reset-to-logo, serial number, and firmware version reads via hid_get_feature_report / hid_send_feature_report.

Because hid_write is a blocking syscall and image uploads consist of many sequential packets, USB I/O runs on its own dedicated worker thread. This prevents the blocking writes from stalling the CORA TCP servers or the web UI event loop.

1a · Device families

Two device generations are supported, with different image formats and HID report layouts.

Gen Model VID PIDs Grid Key image Format
gen2 Stream Deck MK.2 0x0fd9 0x0080, 0x006d, 0x00a5 5×3 (15 keys) 72×72 px JPEG
gen1 Stream Deck Mini 0x0fd9 0x0063, 0x0090, 0x00b3, 0x00b8 3×2 (6 keys) 80×80 px BMP (19 254 B)

1b · The three HID report classes

  • Output report (host → deck) — carries image pixel data for a single key, split into 1024-byte packets. Each packet has a small header describing the key index, chunk number, whether this is the last chunk, and the payload byte count. The image data follows immediately after the header.
  • Input report (deck → host) — reports which keys are currently pressed. The driver polls with a 5 ms timeout and compares against the previous state, emitting down or up events only on state changes (not on every poll).
  • Feature report (bidirectional) — the control channel. Used to set LCD brightness, trigger a logo-reset, and read the device serial number and firmware version. Report IDs and layouts differ between gen1 and gen2 (see section 1e).

1c · Image upload (output reports)

To update a key's image the host sends a sequence of output report packets. Each packet is exactly 1024 bytes (including the HID report ID byte). A header at the front of each packet identifies the target key, the chunk index, whether this is the final chunk, and the number of payload bytes in this packet. Image bytes fill the remainder.

gen2 (JPEG): packet payload area is 1016 bytes; chunk count is ⌈image_len / 1016⌉. Key index is 0-based.

gen1 (BMP, 19 254 bytes): 20 packets total. Key index is 1-based (i.e. the first key is index 1, not 0).

gen2 image output report — 1024 bytes total

gen2 (MK.2): JPEG chunk packet, 1016-byte payload window.

gen1 image output report — 1024 bytes total

gen1 (Mini): BMP chunk packet; key index is 1-based; 20 packets per key image.

1d · Key presses (input reports)

Both generations use HID report ID 0x01 for key state. The button state bytes differ by generation:

  • gen2 — input type byte 0x00 at offset 1; button states begin at data[3] (one byte per key, 0x01 = pressed).
  • gen1 — no input-type prefix; button states begin at data[1].

The driver keeps a copy of the last-seen state. On each hid_read_timeout call it compares the new state against the saved copy and fires keyDown / keyUp events only for keys whose state changed. This means at steady state (no buttons moving) the driver emits nothing, and a button that is held down emits exactly one keyDown.

1e · Feature reports

Feature reports are the control plane. On device open the driver sends a reset-to-logo so any stale image from a previous session is cleared; on close it resets again. All feature report payloads are padded to a fixed length (gen2: 32 bytes; gen1: 17 bytes).

Purpose gen2 payload gen1 payload
Brightness (pct) [0x03, 0x08, pct] (32 B) [0x05, 0x55, 0xaa, 0xd1, 0x01, pct] (17 B)
Reset to logo [0x03, 0x02] (32 B) [0x0b, 0x63] (17 B)
Read serial GET report 0x06: len@1, ASCII@2 GET report 0x03: ASCII@5
Read firmware GET report 0x05: ASCII@6 GET report 0x04: ASCII@5
On open() the driver sends reset-to-logo so stale images clear immediately. On close() it resets again, returning the device to its default logo state.

2 · CORA network protocol

CORA is Elgato's proprietary framed-TCP binary protocol used by its Network Dock hardware. The bridge speaks CORA so the desktop software and third-party tools (e.g. Bitfocus Companion) can discover and drive a USB device over the network without any modification.

Each CORA frame is a fixed 16-byte header followed by an optional variable-length payload. Frames are delimited by a 4-byte magic number (43 93 8A 41) at the start of the header; if the receiver ever loses sync it can scan forward in the stream to re-anchor on the magic. Two TCP ports are used: a primary dock port (5343) for dock-level messages, and a secondary child port for the emulated Stream Deck device (real dock: 20002, our implementation: 5344).

2a · Two-port model

  • Primary "Network Dock" port — TCP 5343
    Dock-level channel: keepalives (every 2 s), device-info responses, and the capabilities packet that announces the child port and connected device geometry. The desktop opens this port first and uses it for the initial handshake.
  • Child port — TCP 5344 (our impl) / 20002 (real dock, dynamic)
    Emulated Stream Deck channel: receives image chunk frames from the desktop, sends key-event frames back, and answers identity/feature probes (serial, firmware). The desktop opens this port after learning its number from the capabilities packet on port 5343.

2b · The 16-byte CORA frame

Every CORA message begins with a 16-byte header. The first four bytes are always the magic 43 93 8A 41, which lets the reader resync if the stream ever desyncs. The header encodes the flags, hidOp, message ID, payload length, and a sequence number used for keepalive/ACK matching.

2c · Flags decoder

The 2-byte flags field in the CORA frame header is a bitfield. The four most significant bits carry the semantic meaning of the frame. Multiple flags can be set simultaneously.

Flag Value Meaning
VERBATIM 0x8000 Payload is a raw HID report — pass through directly to the USB device without interpretation.
REQACK 0x4000 Sender wants an ACKNAK reply from the receiver; used to pace image chunk delivery.
ACKNAK 0x0200 This frame is itself an acknowledgement (or negative-acknowledgement) in response to a REQACK.
RESULT 0x0100 This frame is a response to a request (e.g. a GET_REPORT returning data).

2d · mDNS discovery

The bridge advertises itself on the local network using mDNS so the Elgato desktop software finds it automatically without manual IP configuration. The advertisement uses:

  • Service type: _elg._tcp
  • Instance name: Network Stream Deck
  • TXT record: device type=215, vid=4057 (0x0fd9)
  • Port: 5343 (primary dock port)

Once the desktop resolves the service it opens a TCP connection to port 5343 and begins the CORA handshake described in section 2e.

2e · Connection handshake

The full connection sequence between desktop and bridge proceeds in eight steps:

  1. Desktop discovers the dock via mDNS _elg._tcp.
  2. Desktop opens a TCP connection to port 5343.
  3. Dock immediately sends a keepalive frame ([01 0a 02 .. 01 seq], 32 B) — this is also the "I'm a CORA dock" signal that tells the desktop it is talking to real CORA hardware, not a legacy TCP bridge. The dock continues sending keepalives every 2000 ms.
  4. Desktop → [03 80] device-info GET; dock → RESULT frame with VID 0x0fd9 at byte 12 and PID 0xffff (Network Dock) at byte 14 (both little-endian).
  5. Desktop → [03 1c] capabilities GET (or the dock pushes the capabilities packet unsolicited); dock → capabilities frame announcing connected device geometry, PID, serial string, and the child port number at offset 126 (real dock: 20002, our implementation: 5344).
  6. Desktop opens a TCP connection to the advertised child port.
  7. Child port sends its own keepalive; desktop probes identity, firmware version, and serial number via GET_REPORT frames and VERBATIM probes (gen1: 0xa1 device-info, 0xa4 firmware).
  8. Steady state: image chunks flow desktop→child (ACK-paced); key events flow child→desktop; keepalives flow in both directions on both ports every 2 s. The real desktop software also drops and reopens TCP connections one or two times while probing, which the bridge must tolerate gracefully.

2f · Capabilities packet

The capabilities packet is the most complex CORA frame: it announces everything the desktop needs to know about the emulated device — grid dimensions, PID, serial string, and crucially the child port number to open. It is sent by the dock in response to a [03 1c] capabilities GET (or pushed proactively).

Real-vs-impl deltas: Bytes 2–3 were 0x007c in the real capture; our implementation writes 0x0200. The key-count byte was 0x00 in the capture (the desktop derives the count from rows × cols). The child port at offset 126 was 20002 in the real dock; our implementation uses 5344.

2g · Image delivery & ACK pacing

Image uploads over CORA use the same chunked structure as the USB output report, wrapped in a CORA frame. A gen2 chunk payload begins with a 7-byte header: [0x02, 0x07, keyIdx, isLast, lenLo, lenHi, partIdx] followed by JPEG data — matching the shape of the USB output report so the bridge can forward it directly.

Each chunk frame carries the REQACK flag (0x4000). The bridge must reply with an ACKNAK frame before the desktop sends the next chunk. This back-pressure mechanism is called ACK pacing and ensures the USB write buffer is never overwhelmed. In the real capture, chunk flags are 0xc000 (VERBATIM | REQACK) and hidOp is 0x00 (write) — seen at capture frame #55.

The desktop will not send chunk N+1 until our ACKNAK for chunk N has arrived. This is the primary reason USB I/O runs on a dedicated worker thread: the main thread must stay responsive to send ACKs promptly, while the worker handles the blocking hid_write calls in parallel.

2h · Key events (button-state report)

When a physical button is pressed or released the bridge wraps the HID input report in a CORA frame and sends it to the desktop over the child TCP connection. The full key-state array is included in every event frame — the protocol does not send deltas. The desktop is responsible for comparing successive frames to determine which keys changed.

2i · Keepalive heartbeat

Both the primary (5343) and child (5344) TCP connections exchange keepalive frames independently, every 2 seconds. The sequence number in the keepalive frame increments on each send and wraps at 0xff. The desktop acknowledges primary-port keepalives with a small reply frame ([03 1a seq]); child-port keepalives are acknowledged similarly. Missing keepalives for several cycles signals a dead connection.

3 · End-to-end

Consider a user pressing a button and a macro triggering an image update on that button. In the input direction: the USB worker thread detects a state change in the HID input report, fires a keyDown event to the main thread, which wraps the key-state bytes in a CORA key-event frame and sends it over the child TCP connection to the desktop software. In the output direction: the desktop software decides to update the button image, encodes it as JPEG, splits it into 1016-byte chunks, and sends each chunk as a CORA frame with REQACK set; the bridge receives each chunk, enqueues the HID output report for the USB worker, sends an ACKNAK immediately (so the desktop can release the next chunk), and the USB worker thread calls hid_write to push the pixel data to the physical device. The two directions are fully independent and can proceed simultaneously.
